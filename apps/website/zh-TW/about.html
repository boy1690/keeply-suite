關於 Keeply

Keeply 給每天在共用資料夾上打架的小團隊用。如果你的團隊有人喊過「上週四的版本誰那邊還有？」、找過「真的最終.dwg」、或者眼睜睜看著某個檔案被誰覆蓋——這頁告訴你 Keeply 是誰做的、為什麼做、會走多久。

為什麼我要做一個「版本控制」的軟體

過去這兩年的期間，我們公司承包的一個公共工程案，一名員工負責畫 CAD 施工圖。他畫的圖檔放在公司共用資料夾，跟團隊另外 6-7 個人一起協作。他離職前清理電腦時，剛好誤刪除了其中一個 CAD 圖檔——而那份圖對應的紙本那時還沒掃描歸檔。沒有檔案歷史、沒有變更紀錄、沒辦法回到他誤刪前那一刻。那一份圖，我們只能從零重畫。

我看著這件事發生，第一個念頭不是「我們資安沒做好」也不是「誰該負責」。而是——不是我們笨。是這類軟體本來就沒設計成記得誰做了什麼。NAS、共用資料夾、雲端硬碟，它們的設計目的是讓人同步協作，不是讓人事後追溯。我那時想：工地需要一個檔案管理工具，但不是工程師用的那種。

我自己不是程式背景出身。我深度依賴 AI 寫程式，過程中認識了工程師圈一套叫 git 的東西——它能記得「誰、什麼時候、改了什麼」。那一刻我意識到：這套能力工程師業界用了幾十年，但從來沒有人帶進工地。於是我開始把這層能力，重新設計成適合 6-7 個非工程師共用一份檔案的版本。Keeply 就是這個答案。

在工地我看遍了這些情境——_v3_final.dwg 後面跟著 _v3_really_final.dwg、old/ 資料夾塞 50 個舊版本、團隊群組裡每隔幾天就會出現「誰那邊有上週四的圖」。從那時起我就在想——其他辦公室的小團隊呢？設計師、律師、會計事務所，那些每天 6-7 個人共用一份檔案的場景，會不會也在做同一件事？用命名跟資料夾結構手動模擬檔案歷史。問題不是這些團隊不懂技術，是這個技術從來沒有人翻譯給「不寫 code 的人要一起改一份檔案」的場景聽。

我有時候在想要不要做企業版。目前我還沒看到具體需求，沒看到需求就胡亂設計，只會做出沒有人要的東西。所以我選擇先等——等到有夠多企業用戶來告訴我「你應該加 X」，那時候再認真做。看到那個訊號我會在這頁更新。

Keeply 的 5 條原則

每個人的本機就是備份

你不用相信我們的伺服器。Keeply 裝在你跟團隊每個人的本機，每個人的電腦都有一份完整檔案歷史。6-7 人團隊 = 6-7 份各自獨立的備份。共用 NAS 哪天被誰誤刪、Dropbox 同步出狀況、就算 Keeply 公司明天消失，你還是有同事的本機可以撈回上週四那版。

不改變你們既有的工作流

你們已經有共用資料夾、有命名慣例、有信任的協作流程——這些不需要改。Keeply 不取代 NAS、不取代 Dropbox、不取代 Google Drive。它在每個團隊成員的本機背景留檔案歷史，你們繼續用熟悉的方式協作。沒有人需要重新學工具、沒有人需要被通知「我們改用新的東西了」。

為不寫 code 的團隊設計

你跟團隊每天打開資料夾、找檔案、改檔案——這是大部分人最直覺的工作節奏。Keeply 把檔案歷史接到這個節奏裡：你儲存檔案，Keeply 自動留下檔案歷史；你想回到上週四，點時間軸就好。你不會看到「現在開始要學一套新工具」這種訊息。只要會打開資料夾，就會用 Keeply。

隱私

我們的伺服器看不到你們團隊在改什麼。檔案歷史不上傳、不分析、不賣。我們唯一的商業模式是你付費購買 Keeply 軟體，不是販售團隊資料。一旦你付完授權費，我們跟你的檔案就沒關係了——你關機後我們連你今天有沒有開過 Keeply 都不知道。

5 年後還能讀

Keeply 存檔案歷史用的是公開格式，不是我們發明的封閉檔案。5 年後你想自己寫程式抽出來、想轉去下一個工具、想複製整個歷史備份到別的硬碟——這些你都可以做，不用問我們，也不需要我們的伺服器活著。Keeply 公司明天消失，你的檔案歷史還在原來的位置，可以照常打開。

你會把檔案交給誰？

我是台灣國家高考通過的土木技師，專長營建管理，國立臺灣科技大學（National Taiwan University of Science and Technology）營建工程系畢業。這兩年我在營造廠處理公共工程，工地團隊每天面對的檔案混亂，我是親身經歷的人。

技術那邊就有趣了。我深度依賴 AI 寫程式，一開始我連 GitHub 是什麼都不知道。有一次我開發中的 webapp 大改版，AI 瘋狂改壞我的東西，我當時沒用 git。好在 AI 後來幫我逆向修復回來，我才回頭問：「這種事情根因要怎麼解？」——那是我第一次認識 git。

我以前架 VPS 分銷給全世界，最多 1600 個客戶在用我的服務——所以「在網路上把軟體交到陌生人手上並維運它」對我不是新事。那段經歷教我一件事：客戶不在乎你公司多大，他們在乎你能不能解決問題。Keeply 由我一個人做，意思是沒有產品經理強制我加儀表板、沒有行銷要我寫「AI-powered」、沒有客服腳本擋在你跟我中間。你的訊息會直接到我這裡。對我來說，創業的核心就是幫人解決問題——Keeply 解決的就是工地、辦公室小團隊每天面對的檔案混亂。
