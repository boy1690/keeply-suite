Keeply について

Keeply は、毎日ひとつの共有フォルダーを取り合っている小さなチームのためのツールです。「先週木曜のバージョン、誰のところに残ってる？」と誰かが叫んだことがある。「本当に最終.dwg」を探し回ったことがある。あるいは、自分のファイルが知らないうちに上書きされていた——そんな経験があるなら、このページで、Keeply を誰が、なぜ作ったのか、そしてどれだけ長く使えるのかをお話しします。

なぜ私が「バージョン管理」のソフトを作ったのか

この 2 年ほどの間、私たちの会社が請け負った公共工事の案件で、一人の社員が CAD の施工図を担当していました。彼が描いた図面ファイルは会社の共有フォルダーに置かれ、チームの他の 6〜7 人と一緒に作業していました。退職前にパソコンを整理していたとき、ちょうどその CAD 図面のひとつを誤って削除してしまったのです——しかもその図面に対応する紙の原本は、まだスキャンして保管されていませんでした。ファイル履歴もなく、変更の記録もなく、削除する前の瞬間に戻る方法もありませんでした。その一枚の図面は、ゼロから描き直すしかありませんでした。

それが起きるのを見て、私の最初の考えは「うちのセキュリティが甘かった」でも「誰の責任か」でもありませんでした。そうではなく——私たちが愚かなのではない。この種のソフトは、そもそも誰が何をしたかを覚えるようには設計されていないのだ、ということでした。NAS、共有フォルダー、クラウドドライブ——それらは人が同期して協力するために作られたもので、後から遡って追跡するためのものではありません。そのとき私は思いました。建設現場には、ファイル管理のツールが必要だ。でもエンジニアが使うようなものではない、と。

私自身はプログラミングの出身ではありません。AI に深く頼ってコードを書いていて、その過程でエンジニアの世界にある git というものに出会いました——それは「誰が、いつ、何を変えたか」を覚えていてくれます。その瞬間に気づいたのです。この力をエンジニア業界は何十年も使ってきたのに、誰も建設現場に持ち込んだことがなかった、と。そこで私は、この力を、6〜7 人の非エンジニアがひとつのファイルを共有する場面に合うように作り直し始めました。Keeply がその答えです。

現場で私はこうした場面をすべて見てきました——_v3_final.dwg の後ろに _v3_really_final.dwg が続き、old/ フォルダーに 50 個の古いバージョンが詰め込まれ、チームのグループには数日おきに「先週木曜の図面、誰が持ってる？」が現れる。そのときから私は考え続けていました——他のオフィスの小さなチームはどうだろう？デザイナー、弁護士、会計事務所——毎日 6〜7 人がひとつのファイルを共有しているような場面で、彼らも同じことをしているのではないか？命名とフォルダー構造でファイル履歴を手動で再現する。問題は、こうしたチームが技術を理解していないことではなく、この技術が「コードを書かない人たちがひとつのファイルを一緒に編集する」場面に向けて、誰にも翻訳されてこなかったことなのです。

ときどき企業版を作るべきか考えます。今のところ具体的なニーズは見えていませんし、ニーズが見えないまま闇雲に設計しても、誰も欲しがらないものができるだけです。だから私はまず待つことを選びました——十分な数の企業ユーザーが「X を追加すべきだ」と教えてくれるまで。そのときが来たら本気で作ります。その兆しが見えたら、このページを更新します。

Keeply の 5 つの原則

一人ひとりの手元がバックアップ

私たちのサーバーを信じる必要はありません。Keeply はあなたとチーム全員の手元のパソコンに入り、それぞれのパソコンに完全なファイル履歴があります。6〜7 人のチーム = それぞれ独立した 6〜7 個のバックアップ。共有 NAS がいつか誰かに誤って消されても、Dropbox の同期に問題が起きても、たとえ Keeply の会社が明日消えても、同僚の手元から先週木曜のバージョンを取り戻せます。

今ある仕事の流れを変えない

あなたたちにはすでに共有フォルダーがあり、命名の習慣があり、信頼している協力の流れがあります——それらを変える必要はありません。Keeply は NAS の代わりにもならず、Dropbox の代わりにもならず、Google Drive の代わりにもなりません。チームの一人ひとりの手元の裏側でファイル履歴を残すだけで、あなたたちは慣れたやり方で協力を続けられます。誰もツールを学び直す必要はなく、誰も「新しいものに切り替える」と知らされる必要もありません。

コードを書かないチームのための設計

あなたとチームは毎日フォルダーを開き、ファイルを探し、ファイルを編集します——これがほとんどの人にとって最も直感的な仕事のリズムです。Keeply はファイル履歴をこのリズムにつなぎます。ファイルを保存すると、Keeply が自動でファイル履歴を残し、先週木曜に戻りたければタイムラインをクリックするだけ。「これから新しいツールを学ぶこと」というようなメッセージは出てきません。フォルダーを開けるなら、Keeply は使えます。

プライバシー

私たちのサーバーは、あなたたちのチームが何を変えているか見えません。ファイル履歴はアップロードもせず、分析もせず、売りもしません。私たちの唯一のビジネスモデルは、あなたが Keeply のソフトを購入することであって、チームのデータを売ることではありません。ライセンス料を払い終えれば、私たちとあなたのファイルは無関係です——電源を切れば、あなたが今日 Keeply を開いたかどうかすら私たちには分かりません。

5 年後も読める

Keeply がファイル履歴を保存するのは公開された形式で、私たちが発明した閉じたファイルではありません。5 年後、自分でコードを書いて取り出したい、次のツールに移したい、履歴のバックアップ全体を別のディスクにコピーしたい——どれもできます。私たちに尋ねる必要も、私たちのサーバーが生きている必要もありません。Keeply の会社が明日消えても、あなたのファイル履歴は元の場所にあり、いつも通り開けます。

あなたは誰にファイルを預けますか？

私は台湾の国家試験に合格した土木技師（Professional Civil Engineer）で、専門は施工管理、国立台湾科技大学（National Taiwan University of Science and Technology）の建設工学科を卒業しました。この 2 年は建設会社で公共工事を担当していて、現場のチームが毎日向き合うファイルの混乱を、私は身をもって経験しています。

技術のほうは面白い話です。私は AI に深く頼ってコードを書いていて、最初は GitHub が何かすら知りませんでした。あるとき開発中の webapp を大きく作り直していたら、AI が私のものをめちゃくちゃに壊してしまい、そのとき私は git を使っていませんでした。幸い AI が後から逆にたどって直してくれて、そこでようやく振り返って尋ねたのです。「こういうことは根本からどう解決するんだ？」——それが私が初めて git を知った瞬間でした。

私は以前、世界中に VPS を再販していて、最大で 1600 人の顧客が私のサービスを使っていました——だから「インターネットで見知らぬ人にソフトを届けて運用し続ける」ことは私にとって新しいことではありません。あの経験が教えてくれたことがひとつあります。顧客はあなたの会社の大きさを気にしない、あなたが問題を解決できるかどうかを気にする、ということです。Keeply を一人で作っているということは、ダッシュボードを追加しろと強いるプロダクトマネージャーもいなければ、「AI-powered」と書けと言うマーケティングもいなければ、あなたと私の間に立つサポートの台本もない、ということです。あなたのメッセージは直接私のところに届きます。私にとって、起業の核心は人の問題を解決することです——Keeply が解決するのは、建設現場やオフィスの小さなチームが毎日向き合うファイルの混乱です。
