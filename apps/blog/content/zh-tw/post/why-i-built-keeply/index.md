  - q: Keeply 跟 Git 差在哪裡？
    a: Git 是給寫程式的人設計的：每存一次都要主動下指令、要學分支跟合併概念、要管 repository。Keeply 把同樣的「版本歷史」做成一般檔案工作者也用得上的樣子：拖一個資料夾進來、重要時刻按「儲存版本」存一版（或開背景自動儲存每 15/30/60 分），版本歷史就出來——不用下指令、不用懂分支合併。
  - q: 為什麼決定做 Keeply？
    a: 在營建業現場看了太多次同樣的事——同事一次未存檔覆蓋失去 6 週的 CAD 圖、工地師傅照上週舊版施工結果混凝土打掉重做。市面上的版本管理工具都假設使用者願意學 Git，但建築師、設計師、會計師、律師、研究生都不是要學 Git，他們要的是「我做事的時候在背景有人幫我留版本」。
  - q: Keeply 想解決什麼問題？
    a: 一個被忽視的中間層：檔案級的故意存檔版本歷史。雲端同步解的是檔案存在、Time Machine 解的是整顆磁碟還原、Word 修訂只解這份檔案這次 session 的編輯痕跡。「3 個月前禮拜二我特地存的那版」這層缺口沒有工具佔位，多數人靠手動加 _v7 _final 對抗——撐不過半年。
  - q: 適合誰、不適合誰？
    a: 適合：建築師、設計師、律師、會計師、研究生、AI 配對寫程式的開發者、任何在共用資料夾中要追蹤誰改了什麼的人。不適合：純文字 code 用 Git/GitHub、強制走 SharePoint/OneDrive 為唯一文件源的企業 IT 政策、檔案全部在雲端原生工具（Google Docs/Notion）的人。我們沒打算搶 Git 的市場、也不想取代雲端同步——Keeply 只做被遺漏的那一層。

我這幾年陪著營建業的工程師工作。他們很多人 50、60 歲，電腦不是他們最自在的工具，但每天的工程圖、變更單、合約都得在電腦上跑。共用資料夾是他們協作的主場：一個 NAS、一群人、N 份檔案、隨時在改。

[失控的場景](/zh-tw/post/autocad-wrong-version-crew/)我看過太多次：設計改了新版本寄到辦公室、收信的人存進 NAS、沒通知工地。工地師傅那天在用的是上禮拜的舊圖。混凝土已經灌下去，尺寸不對，要打石、重埋、工期延兩天。沒人做錯，但有人付代價。

## 我選的那條岔路

我自己用 git 很順。軟體工程師都會，外面甚至有付費課程教你怎麼用。每次發現有問題，我會 commit、branch、reset，工具像第二隻手一樣自然。

但我一說「試試 git」給營建師傅，每次都換來一張困惑的臉。git 不是給他們的：黑底白字的終端機、merge conflict 提示、HEAD 指標——每一個概念都是擋路的牆。我夾在中間：自己用得順、客戶用不來、共用資料夾繼續每週製造混凝土打掉的故事。

教他們學 git 比較便宜。做一個他們不用學的工具難。我選了難的那條。

## 我犯過的錯（之一）

第一版的 Keeply 我塞了太多功能。我想：營建業要、設計師也要、會計事務所也會用，每個情境我都想接住，結果做出來像瑞士刀，每個功能都有，但沒人用得順。

後來我刪了一輪、又一輪。

現在每個新功能要進 Keeply，我先問自己 3 題：營建現場真的會用嗎？我那群 60 歲的師傅會打開嗎？刪掉誰會在意？任一 no = 不做。功能少不是 錯誤，是設計選擇。

## 為什麼寫這篇

這篇講**初心**。

Keeply 不是要在版本管理工具圈裡贏 git、贏 SVN、贏 Mercurial。那場仗開發者們贏了二十年。Keeply 是給每天**不打開 git** 但每天**打開資料夾**的人：營建師傅、設計師、律師、會計、學生、接案者。

如果你正是這群人之一，你看 [共用資料夾的檔案版本問題](/zh-tw/post/hidden-cost-shared-folders/) 那篇就會懂：你不是不會整理檔案，是工具把整理檔案的責任全推給你的記憶力。
