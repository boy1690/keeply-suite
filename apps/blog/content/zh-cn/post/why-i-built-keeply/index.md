faq_schema:
  - q: Keeply 跟 Git 差在哪里？
    a: Git 是给写程序的人设计的：每一次保存要主动下命令、要学分支跟合并概念、要管 repository。Keeply 把同样的「版本历史」做成一般文件工作者也用得上的样子：拖一个文件夹进来、重要时刻按「保存版本」存一版（或开后台自动保存每 15/30/60 分），版本历史就出来——不用下命令、不用懂分支合并。
  - q: 为什么决定做 Keeply？
    a: 在营建业现场看了太多次同样的事——同事一次未保存覆盖失去 6 周的 CAD 图、工地师傅照上周旧版施工结果混凝土打掉重做。市面上的版本管理工具都假设用户愿意学 Git，但建筑师、设计师、会计师、律师、研究生都不是要学 Git，他们要的是「我做事的时候在后台有人帮我留版本」。
  - q: Keeply 想解决什么问题？
    a: 一个被忽视的中间层：文件级的故意保存版本历史。云端同步解的是文件存在、Time Machine 解的是整块硬盘还原、Word 修订只解这份文件这次 session 的编辑痕迹。「3 个月前周二我特地保存的那版」这层缺口没有工具占位，多数人靠手动加 _v7 _final 对抗——撑不过半年。
  - q: 适合谁、不适合谁？
    a: 适合：建筑师、设计师、律师、会计师、研究生、AI 配对编程的开发者、任何在共享文件夹中要追踪谁改了什么的人。不适合：纯文本 code 用 Git/GitHub、强制走 SharePoint/OneDrive 为唯一文件源的企业 IT 政策、文件全部在云端原生工具（Google Docs/Notion）的人。我们没打算抢 Git 的市场、也不想取代云端同步——Keeply 只做被遗漏的那一层。

我这几年陪着营建业的工程师工作。他们很多人 50、60 岁，电脑不是他们最自在的工具，但每天的工程图、变更单、合同都得在电脑上跑。共用文件夹是他们协作的主场：一个 NAS、一群人、N 份文件、随时在改。

[失控的场景](/zh-cn/post/autocad-wrong-version-crew/)我看过太多次：设计改了新版本寄到办公室、收信的人存进 NAS、没通知工地。工地师傅那天在用的是上礼拜的旧图。混凝土已经灌下去，尺寸不对，要打石、重埋、工期延两天。没人做错，但有人付代价。

## 我选的那条岔路

我自己用 git 很顺。软件工程师都会，外面甚至有付费课程教你怎么用。每次发现有问题，我会 commit、branch、reset，工具像第二只手一样自然。

但我一说「试试 git」给营建师傅，每次都换来一张困惑的脸。git 不是给他们的：黑底白字的终端、merge conflict 提示、HEAD 指针——每一个概念都是挡路的墙。我夹在中间：自己用得顺、客户用不来、共用文件夹继续每周制造混凝土打掉的故事。

教他们学 git 比较便宜。做一个他们不用学的工具难。我选了难的那条。

## 我犯过的错（之一）

第一版的 Keeply 我塞了太多功能。我想：营建业要、设计师也要、会计事务所也会用，每个场景我都想接住，结果做出来像瑞士刀，每个功能都有，但没人用得顺。

后来我删了一轮、又一轮。

现在每个新功能要进 Keeply，我先问自己 3 题：营建现场真的会用吗？我那群 60 岁的师傅会打开吗？删掉谁会在意？任一 no = 不做。功能少不是 错误，是设计选择。

## 为什么写这篇

这篇讲**初心**。

Keeply 不是要在版本管理工具圈里赢 git、赢 SVN、赢 Mercurial。那场仗开发者们赢了二十年。Keeply 是给每天**不打开 git** 但每天**打开文件夹**的人：营建师傅、设计师、律师、会计、学生、接案者。

如果你正是这群人之一，你看 [共用文件夹的文件版本问题](/zh-cn/post/hidden-cost-shared-folders/) 那篇就会懂：你不是不会整理文件，是工具把整理文件的责任全推给你的记忆力。

下一个版本见。
