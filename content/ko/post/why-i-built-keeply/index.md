첫 번째 얼리 테스터가 NAS 폴더를 열어 자기 파일을 보려 했다. 폴더 안에 있던 건 `objects/`, `pack/`, `HEAD`. 그가 저장한 디자인 파일은 없었다. 그가 메시지를 보냈다: 「내 파일이 어디로 갔지?」

그 순간, 무엇을 잘못했는지 알았다.

우리는 「백업」을 핵심 개념으로 골랐다. 엔지니어 머리에서 백업은 OK: 압축됨, 인코딩됨, 직접 보이지 않음. 하지만 사용자에게 **「백업」의 반대말은 「백업 없음」이 아니라 「못 찾음」이다**. 폴더를 열어 자기 것이 안 보이면 = 신뢰 깨짐. 기술이 안전하다고 말해도, 눈이 안전하지 않다고 말하면 안전하지 않다.

이게 Keeply 의 첫 번째 전환점. 나중에 정식 [ADR-001](https://github.com/boy1690) 이 되었다: 「백업」을 핵심 비유에서 빼고, 「프로젝트 위치」로 전환. 한 단어 차이로, 데이터 구조 전체가 바뀜.

## 내가 선택한 갈림길

두 갈래가 있었다. 사용자를 dev 로 교육시키거나 (`objects/` 가 pack file 인 것, `HEAD` 가 포인터인 것 배우게 하기), 도구를 사무실 언어로 만들거나 (「버전 저장」「버전 기록」「복원」).

사용자 교육이 더 싸다. 도구를 만드는 게 더 어렵다. 나는 후자를 골랐다.

Keeply 의 [미션](https://github.com/boy1690) 은 그 후 한 문장이 되었다: 「**비기술자가 사무실 언어로 파일 버전을 관리할 수 있게, Git 의 존재를 알 필요가 전혀 없도록.**」 UI 에 commit, branch, HEAD, stash 가 안 나옴, 비유로도 안 됨. 아래에서 git2 엔진을 쓰지만, 그건 내 문제이지 당신 문제가 아님.

## 내가 한 실수 (중 하나)

모든 설계 판단이 옳은 건 아니다. 올해 4 월에 Free / Team 차별화 전략 초안을 상위 모델에 부탁했다. 돌아온 건 530 줄: 5 종 용도 할당량, watermark 증거, RFC 3161 타임스탬프, 5 가지 복잡한 업그레이드 트리거.

전부 거부했다.

이유: watermark 는 대만에서 법적 증거가 아님 (정식 공문서가 그것). NAS 위 여러 폴더는 물리적으로 multi-vault 와 동등—숫자 제한에 의미 없음. RFC 3161 타임스탬프는 대만 사용자에게 실질적 셀링 포인트 없음 (우체국 보존 증명이나 공증으로 감). **이 feature 들은 이론을 충족할 뿐, 실제 사용자에겐 도움 안 됨.**

지금은 모든 spec 결정이 3 가지 질문을 통과한다: 사용자가 원하나? 실제 시나리오에서 의미 있나? 빼면 누가 신경 쓰나? 어느 하나라도 「no」 = ship 안 함.

## 왜 이걸 쓰나

마케팅 글이 아니다. **투명성** 이다: 내가 이 도구를 만든 이유, 한 실수, 지키는 원칙.

클라이언트 데이터, 디자인 파일, 계약서를 5 년 10 년 도구에 맡길 거라면, 만든 사람이 어떻게 생각하는지 알아야 한다. Keeply 가 영원히 옳을 거라고 보장 못 함, 하지만 보장할 수 있는 건: 모든 결정은 적어둠, 모든 실수는 reframe 됨, 모든 over-engineered 아이디어는 거부됨.
