Engineer A Cursor खोलते हैं और AI को एक bug ठीक करने को कहते हैं। AI पूरा करता है। Code नहीं चलता। वह AI को फिर से ठीक करने को कहते हैं। AI तीसरी फ़ाइल छूता है। फिर भी टूटा। यह पाँचवीं को एडिट करता है। अब Engineer A को यक़ीन नहीं कि AI ने कौन-कौन सी फ़ाइलें बदली हैं।

इस वक़्त आप शायद सोच रहे हैं: रुको, उस state पर वापस जाओ जो कम से कम कुछ देर पहले चल रही थी।

समस्या यह है: **आपको कैसे पता कि कौन-सा वर्शन वह था जो चल रहा था?**

## AI के overshoot का पल कैसा दिखता है? {#ai-overshoot}

आप vibe coding कर रहे हैं। आप AI को एक goal देते हैं। AI एक chunk लिखता है।

चलाइए। OK।

अगले राउंड में, आप कहते हैं "एक और feature जोड़ो।" AI 3 फ़ाइलें छूता है। चलाइए — error।

आप कहते हैं "उस error को ठीक करो।" AI 5 फ़ाइलें छूता है, config एडिट करता है, एक helper function जोड़ देता है जो आपने कभी नहीं माँगा। चलाइए — और errors।

![AI agent chat window बनाम आपके कंप्यूटर पर असल में बदली गई फ़ाइलों की संख्या](image-1.svg)

AI अब भी आत्मविश्वास से ठीक करता रहता है। **वह ख़ुद से नहीं कहेगा "शायद मैंने इसे बिगाड़ दिया।"**

उसकी memory सिर्फ़ मौजूदा context window है। **उसे नहीं पता कि 5 prompts पहले आपका code ठीक था।** पर आपके कंप्यूटर पर मौजूद फ़ाइलों को पता है। बशर्ते कोई याद रखे।

## एक action: Timeline खोलिए, आख़िरी चलने वाले पॉइंट पर क्लिक कीजिए {#one-action}

### स्टेप 1: Keeply Timeline खोलिए

बाएँ sidebar का पहला tab। आपको आज के सारे बदलाव दिखेंगे, समय के क्रम में।

### स्टेप 2: वह आख़िरी पॉइंट ढूँढिए जहाँ code "अभी भी चल रहा था"

Timeline पर हर entry या तो Keeply का ऑटो सेव पॉइंट है या एक पल जिसे आपने मैनुअली मार्क किया। हर पॉइंट खोलकर अंदर के बदलाव देखिए, और वह वर्शन ढूँढिए जिसे आप "तब OK टेस्ट किया था" के रूप में याद करते हैं।

आम तौर पर 30-60 मिनट पहले। AI के टेढ़ा होने से पहले का आख़िरी टेस्ट।

![Keeply Timeline ज़ूम-इन: हर file note timestamp + बदली गई lines + आपका पहले का टेस्ट रिकॉर्ड दिखाता है](image-2.svg)

### स्टेप 3: उस entry पर right-click कीजिए, Restore चुनिए

पूरा फ़ोल्डर 30 सेकंड में उस पॉइंट पर वापस आ जाता है। **सारी फ़ाइलें, पूरी directory tree, हर config — सब साथ वापस जाते हैं।** सिर्फ़ एक फ़ाइल नहीं।

इसमें वह helper function शामिल है जो AI ने चुपके से डाला, वह config जो उसने एडिट किया, वह `.env` जो उसे छूना नहीं चाहिए था। **सब वापस।**

फिर आप चलाते हैं। चलता है।

![रिस्टोर से पहले बनाम बाद: file tree + टेस्ट चलाने की हरी रोशनी](image-3.svg)

पूरी प्रक्रिया एक मिनट से कम लेती है। **आपको याद रखने की ज़रूरत नहीं कि AI ने कौन-सी फ़ाइलें छुईं। Keeply ने सब याद रखा।**

## AI ख़ुद को क्यों रिस्टोर नहीं करेगा {#ai-doesnt-rollback}

AI agents **आगे चलाने** के लिए डिज़ाइन हैं। वे एक prompt पाते हैं, एक एडिट बनाते हैं। वे रुककर पीछे देखकर यह नहीं पूछेंगे कि "क्या उस आख़िरी राउंड ने प्रोजेक्ट को और बिगाड़ दिया?"

वह ज़िम्मेदारी AI के पास नहीं बैठती। यह एक architectural सीमा है।

ज़िम्मेदारी आप पर बैठती है: **आपको बैकग्राउंड में चलने वाला एक सुरक्षा-जाल चाहिए।** AI को जितना चाहे दौड़ने दीजिए, क्योंकि आप उसे वापस खींच सकते हैं।

Keeply वह जगह नहीं लेने आया जहाँ आप code लिखते हैं। यह इसलिए है कि जब आप vibe coding कर रहे हों, आपको पीछे जाने के लिए memory पर निर्भर न रहना पड़े। Memory AI के फ़ाइल एडिट करने की रफ़्तार से हार जाती है।
